Detailed Description

Multi-threaded server that maintains and distributes tracker files.

Authors
Matthew Lindner, Xiao Deng

Centralized, multi-threaded server whose purpose is to distribute tracker files to clients who what to share files on a P2P network.

The server can process four types of requests:

  1. createtracker
  2. updatetracker
  3. LIST
  4. GET

Once the server processes a single request from a client, it closes the connection to that client. Each client is handled in its own thread. This allows the server to handle multiple clients at a single time.

COMPILE

gcc server.c -o server.out -lnsl -pthread -lcrypto

void * client_handler ( void *  index)

When a client connects to the server, it's thread of execution is handled by this function. A command (createtracker, updatetracker, LIST, GET) will be read from the client's socket, and processed.

Parameters
indexIndex in the clients array where the new client is stored.

Read data from the peer's socket, store it in m_buf. Compare the string stored in m_buf to see if it is any of the four commands. And then serve the peer.

CREATETRACKER Command

If client did not send the correct number of arguments, send a "createtracker fail" protocol message.

The create tracker command is broken up into 7 words: createtracker, filename, filesize, description, md5, ip, and port. We are interested in the last 6.

If this tracker file already exists, send a "createtracker ferr" protocol message.

Otherwise, since this tracker file does not exist, create the tracker file. Start by creating an empty file.

Let the client know that the creation was successful with a "createtracker succ" protocol message.

If there was a problem creating the file, send the user a "createtracker fail" protocol message.

UPDATETRACKER Command

The update tracker command is broken up into 6 words: updatetracker, filename, start byte, end byte, ip, and port. We are interested in the last 5.

If this tracker file does not exist, send a "createtracker ferr" protocol message.

Otherwise, append the new chunk data to the end of the tracker file.

Let the client know that the update was successful with a "updatetracker succ" protocol message.

If there was a problem updating the file, send the user a "updatetracker fail" protocol message.

REQ LIST Command

First, opens the "Tracker Files" folder, and counts the number of files.

For each file in the "Tracker Files" folder, stores the tracker file's: Filename, filesize, and md5.

Sends each tracker file info, indexed by a number.

GET Command

First, the server checks to make sure that the requested tracker file already exists.

Next, it opens the file, and copies it's contents.

It then sends the peer the tracker file, copied into a buffer.

Finally, it includes the md5 sum of the tracker file itself, and appends it to the end of the "GET" protocol footer.

If the tracker file could not be opened, the server sends the peer a "GET invalid" protocol error message.

If the file does not exist, the server sends the peer a "GET invalid" protocol error message.

Closing the connection to the peer.

Once the peer's request has been handled, the server closes that socket, indicates that there is a new opening in the client array, and ends the thread.

int findClientArrayOpening ( )

Returns an index indicating where a new client can be stored in the clients array. Checks each element of the array sequentially until an unused slot is located.

Returns
Index (0 - MAX_CLIENT -1) of the clients array where new client will be stored, -1 if array is full (server is currently serving the maximum amount of clients).
int main ( )

When server.out is executed, the server will listen for peers on a stream socket. When a peer connects, if the server is not already at capacity (MAX_CLIENT), a thread will be spun off to handle the peer. Multiple peers can be handled at once. The server will then read a command from that peer's socket, and process that command. Once that single command has been processed, the server disconnect from the peer.

Create a stream socket. Server will listen on this socket for peers.

Bind the socket to an internet port.

Listen for clients. MAX_CLIENT = length of queue for pending connections.

Initialize the client array so each element is marked as unused.

While running, the server will allow up to a predefined number of peers connect at once. When a peer tries to connect, if the server is not at capacity, the peer will be placed in the clients array, and a thread will be spun off to process that peer. If the server is already at full capacity, the peer will have to wait for an opening.

void setUpClientArray ( )

Initializes each peer in the clients array.

Each m_index is set to i, the index of the array (0 - MAX_CLIENT). Each m_peer_socket is set to -1 to indicate that this peer is currently not being used.
